### H1 — `master_id_pool` : O(n_base) Strings retenues tout le run, éliminable entièrement

- **Fichier** : `pipeline.rs:504` (déclaration), `:554` (`mids = Vec::with_capacity(plan.n_base)`), `:667` (`mids.extend(batch_mids)`), `:702` (insert), `:710` (seul usage réel).
- **Statut** : vivant, chemin principal, toutes tailles — c'est LE poste RAM dominant avec H2.
- **Constat** : pour chaque entité, un `Vec<String>` de `n_base` master_ids (`"E12345-" + 13 digits` ≈ 19 octets heap + 24 octets header, ~56 o/entrée avec bins allocateur) est accumulé puis inséré dans `master_id_pool`, qui vit jusqu'à la fin du run. Le commentaire ligne 701 dit « for FK remap and dup mid cloning » mais c'est faux pour la première moitié : le FK remap utilise `fk_pools`, jamais `master_id_pool`. Le seul lecteur est la section dups (`mids_ref`, ligne 710), qui n'accède qu'aux indices `last_offset + idx` — c'est-à-dire uniquement la plage du dernier batch. À 200M records base : ~11 Go retenus pour rien. En prime, `Vec::with_capacity(plan.n_base)` pré-alloue 200M × 24 o = 4,8 Go de pointeurs dès le début de l'entité.
- **Levier** : supprimer `master_id_pool` et `mids` entièrement. Dans la closure dup (ligne 758), remplacer `mslice[last_offset + idxs.value(j) as usize].clone()` par `format!("{}-{}", prefix, pad_string(last_offset + idxs.value(j) as usize))` — le master_id est une fonction pure de (prefix, index), exactement comme record_id l'était avant le fix IdPools. `prefix` est déjà disponible dans le scope de l'entité (capturer un `&str`/clone léger dans la closure).
- **Bit-identique** : **garanti par construction** — égalité de fonction pure : `mids[i] == format!("{prefix}-{}", pad_string(i))` pour tout i (c'est exactement ainsi que `batch_mids` est construit ligne 636-638).
- **Priorité** : **P0**. Confiance : très haute. Effort : faible (~15 lignes). C'est le raffinement du backlog « plafonner master_id_pool » — pas besoin de plafonner, on supprime.
- **Mesure isolée** : A/B peak RSS (`Get-Process`/`/usr/bin/time` équivalent ou compteur maison) sur `--size 50M`, + checksum SHA256 des fichiers IPC de sortie (doivent être identiques octet pour octet).

### H2 — Comptage GT : `FxHashMap<String, u32>` avec une String allouée par master distinct

- **Fichier** : `gt.rs:51` (champ `counts`), `:79-89` (`push_batch`), `:163` (lecture), `:195` (`counts.len()` = n_masters).
- **Statut** : vivant, chemin principal, toutes tailles — second poste RAM dominant.
- **Constat** : chaque master distinct (= chaque record base, ils sont tous distincts) coûte une entrée `FxHashMap<String, u32>` : String 24 o header + ~19-32 o heap + u32 + overhead buckets ≈ 60-80 o/master. À 200M base : 12-16 Go. De plus `counts.entry(mid.to_string())` alloue la String même quand la clé existe déjà (l'API `entry` exige une clé owned).
- **Levier** — trois options par ordre de gain :
  - **(a) micro, gratuit** : remplacer le pattern `entry(mid.to_string())` par `if let Some(c) = counts.get_mut(mid) { *c += 1 } else { counts.insert(mid.to_string(), 1); }` — épargne l'alloc sur les occurrences répétées (dups). Bit-identique garanti.
  - **(b) dense** : les suffixes de master_id sont denses `0..n_base` par préfixe d'entité. Remplacer la map par `FxHashMap<String /*prefix*/, Vec<u8 /*saturating*/>>` — vecteur indexé par suffixe, incrément saturant (on ne teste que `> 1`). 200M masters → ~200 Mo au lieu de 12-16 Go. `n_masters` = nombre d'entrées non nulles (ou somme des `n_base`, voir (c)). Bit-identique sur le fichier de sortie (la classification ne dépend que de count>1) ; à valider par A/B checksum par prudence (parsing du suffixe).
  - **(c) structurel, le plus fort** : ne pas compter du tout. Le pipeline *sait* quels mids sont dupliqués (`dup_mids_buf`). Étendre l'API de `GtAccumulator` pour recevoir l'ensemble des dup-mids (petit : O(n_dups), quelques % du dataset) ; classification = `mid ∈ dup_set → exact_dup`. `n_masters` devient arithmétique (Σ n_base — chaque record base a un mid unique). RAM GT ≈ O(n_dups) au lieu de O(n_base). PAS bit-identique *a priori* trivialement (équivalence logique : count>1 ⟺ mid dupliqué ⟺ mid ∈ dup_set) — théorème simple mais A/B checksum obligatoire, et attention à ce que les tests unitaires de `gt.rs` qui poussent des mids arbitraires restent valides (l'API change).
- **Priorité** : **P0** (option b ou c). Confiance : haute. Effort : (a) trivial, (b) moyen, (c) moyen+ (change l'API interne).
- **Mesure isolée** : même protocole que H1 (A/B RSS + checksum GT file). H1+H2 ensemble devraient ramener le 200M sous ~8-10 Go.

### H3 — Conversion Parquet : row group illimité ⇒ tout le dataset bufferisé en RAM

- **Fichier** : `pipeline.rs:965` — `set_max_row_group_row_count(Some(usize::MAX / 2))`.
- **Statut** : conditionnel — uniquement `--output-format parquet`.
- **Constat** : `ArrowWriter` ne flush un row group sur disque qu'à l'atteinte de `max_row_group_row_count`. Avec `usize::MAX/2`, le row group ne se ferme jamais avant `close()` : toutes les column chunks compressées (et les buffers d'encodage) restent en RAM pendant toute la conversion. À 200M records, c'est plusieurs Go de données compressées ZSTD résidentes — la conversion Parquet annule une partie du bénéfice du streaming IPC. Noter que le sink GT Parquet (`gt.rs:223-227`) n'a pas ce problème (défaut 1M lignes).
- **Levier** : fixer un row group raisonnable (p.ex. 1_000_000 lignes, cohérent avec le défaut parquet-rs) ou supprimer l'appel.
- **Bit-identique** : **NON garanti au niveau fichier** — les frontières de row groups changent le binaire Parquet (les *valeurs* restent identiques). A/B obligatoire par checksum du *contenu* (relecture + hash des colonnes), pas du fichier.
- **Priorité** : P1. Confiance : haute sur le mécanisme (à confirmer sur pièce avec la version du crate parquet du lockfile). Effort : trivial.
- **Mesure isolée** : run `--output-format parquet --size 20M`, RSS échantillonné pendant la phase 3b uniquement, avant/après.

### H4 — Pool HN mono-slice : zero-copy Arrow épingle le batch complet de 500K lignes

- **Fichier** : `pipeline.rs:615-619` (accumulation `rb.slice(0, take)`), `:683-698` (matérialisation).
- **Statut** : vivant ; l'ampleur dépend de `hn_max_pool` vs `BATCH_SIZE`.
- **Constat** : `rb.slice(0, take)` est zero-copy — la slice retient les buffers du batch *entier* (500K lignes × toutes colonnes). Quand `hn_slices.len() > 1`, le `concat` final compacte et libère. Mais quand `hn_max_pool ≤ batch_n` (cas typique : counts HN modestes ⇒ une seule slice sur le premier batch), la branche `hn_slices.len() == 1` (ligne 685-686) stocke **la slice elle-même** dans `hn_pools` — qui vit jusqu'à la fin du run. Résultat : un batch complet de 500K lignes épinglé par entité, pour ne servir que `hn_max_pool` lignes. Sur un schéma multi-entités à colonnes larges, plusieurs centaines de Mo à quelques Go retenus inutilement.
- **Levier** : dans le cas mono-slice avec `take < rb.num_rows()`, forcer une compaction (p.ex. `arrow::compute::concat(&[slice])` sur chaque colonne, ou `take` kernel) pour matérialiser des buffers propres avant l'insert dans `hn_pools`.
- **Bit-identique** : **garanti par construction** — mêmes valeurs, seule la matérialisation mémoire change.
- **Priorité** : P1. Confiance : haute. Effort : faible.
- **Mesure isolée** : A/B RSS à `--size 10M` sur un domaine multi-entités (p.ex. ecommerce), checksum sorties.

### H5 — Pools HN : dimensionnement au max global + pools construits pour des entités jamais ciblées

- **Fichier** : `pipeline.rs:524-533` (`hn_pool_sizing` → `hn_max_pool` global), `:615-619` / `:683-698` (accumulation inconditionnelle).
- **Statut** : vivant.
- **Constat** : deux gaspillages distincts. (1) `hn_pool_sizing` calcule bien un max *par type d'entité*, mais la ligne 533 l'écrase en un max *global* appliqué à toutes les entités — une entité avec un petit count HN se voit allouer le pool de la plus gourmande. (2) Les slices HN sont accumulées pour *toutes* les entités, y compris celles qui n'apparaissent dans aucun `hard_neg_types` — leur pool n'est jamais lu (`hn_pools.get(&hn_cfg.entity_type)` ligne 835) et reste résident jusqu'à la fin.
- **Levier** : (2) sauter l'accumulation `hn_slices` quand `hn_pool_sizing.get(plan.name.as_str())` est absent — **bit-identique garanti** (pool jamais lu ; vérifier qu'aucun autre lecteur n'existe). (1) utiliser `hn_pool_sizing.get(...)` par entité au lieu du max global — **PAS bit-identique** : la taille du pool change les lignes échantillonnées par `generate_hard_negatives` ⇒ A/B checksum obligatoire, sortie différente attendue ⇒ à traiter comme un changement de comportement assumé, pas un pur fix perf.
- **Priorité** : P2 (le (2) d'abord, gratuit et sûr). Confiance : haute. Effort : faible.

### H7 — Fichier draft GT : ~40+ octets/ligne écrits puis relus, largement reconstructibles

- **Fichier** : `gt.rs:13-19` (schéma draft 3 colonnes Utf8), `:91-99` (écriture), `:146-190` (relecture).
- **Statut** : vivant (architecture du fix GT streaming). Coût : I/O disque ~10-16 Go aller-retour à 200M + sérialisation IPC.
- **Constat** : le draft stocke `record_id` (pourtant strictement séquentiel dans l'ordre d'écriture : `R-` + rang global), `entity_type` (constant par batch) et `master_id` (fonction pure de (prefix, index) hors HN/CANARY) en Utf8 plein. Presque tout est reconstructible au `finish()`.

Se marie naturellement avec H2(b)/(c).
- **Bit-identique** : **garanti par construction** si la reconstruction est exacte (mêmes fonctions pures) — à verrouiller par A/B checksum du GT final quand même.

### H8 — Conditions de colonnes : matérialisation `Vec<String>` de la colonne entière par condition

- **Fichier** : `entity_gen.rs:206-210` (`array_to_strings`), appelé par `build_condition_mask` (`:158`, `:168`) ; `array_to_f64s` (`:212-220`) même motif en numérique.

### H13 — `StringBuilder::with_capacity(n, X)` : 2ᵉ argument traité comme largeur par ligne alors que c'est la capacité totale en octets

- **Fichiers** : motif systématique, ~18 sites.
  - `column_gen.rs:203` (`gen_date`, `(n, 10)`), `:215` (`gen_datetime`, `(n, 19)`), `:314` (fallback `_id`, `(n, 12)`).
  - `fast_template.rs:638` (`gen_suffix`, `(n, 4)`), `:656`, `:671`, `:682`, `:699`, `:708`, `:724`, `:744`, `:753`, `:763` (pools statiques divers), `:1061`, `:1069`, `:1078`, `:1086`, `:1094` (volume/issue/option1-3).
  - `pipeline.rs:379` (`missing_pattern`, `(n, 32)`) — repéré rétroactivement, même motif.
- **Statut** : vivant, chaque batch, chaque colonne concernée.
- **Constat** : la signature Arrow est `with_capacity(item_capacity, data_capacity)` où `data_capacity` est le total d'octets du buffer de valeurs. Passer `10` pour 500 000 dates signifie : buffer initial de 10 octets, puis ~20 doublements successifs avec memcpy à chaque fois (~2× la taille finale copiée, pics transitoires 1,5× le buffer pendant le realloc). Les sites corrects du crate (`buf_gen.rs:30,51,76`, `gt.rs:151`) passent bien `n * width` — la moitié du code a compris la signature, l'autre non.
- **Levier** : multiplier par `n` partout : `with_capacity(n, n * avg_width)`. Pour les pools statiques, `avg_width` = largeur max du pool (déjà la valeur passée aujourd'hui).
- **Bit-identique** : **garanti par construction** (capacité d'allocation, aucune valeur ne change).
- **Priorité** : P2. Confiance : haute sur le mécanisme, moyenne sur l'ampleur (les doublements sont amortis O(n) — c'est du memcpy évitable, pas un désastre). Effort : trivial (~18 lignes).
- **Mesure isolée** : `examples/profile_builders.rs` — construire une colonne date de 500K avec capacité 10 vs n*10, chronométrer 20 itérations. Trancher AVANT de généraliser le fix si le gain est < bruit.

### H14 — `format!`/`to_string` par ligne dans gen_date, gen_datetime, fallback `_id`, volume/issue

### H19 — ⚠ BUG DÉTERMINISME (pas perf) : `language_mix` itère une `HashMap` std, ordre aléatoire par processus

- **Fichier** : `noise/addresses.rs:15-29` (`FR_TO_EN: LazyLock<HashMap<...>>`), `:76-80` (`for (fr, en) in FR_TO_EN.iter()`).
- **Statut** : vivant dès qu'un schéma utilise le bruit `addresses`/`language_mix`.
- **Constat** : `std::collections::HashMap` a un ordre d'itération aléatoire **par processus** (RandomState). Or la boucle consomme le RNG conditionnellement (`if result.contains(fr) && rng2.next_f64() < 0.3`) : l'ordre de visite des paires détermine quelles substitutions sont testées en premier ET combien de tirages RNG sont consommés. Deux runs avec le même `--seed` peuvent donc produire des sorties différentes sur les colonnes touchées par `language_mix` — violation directe de la garantie de déterminisme du générateur. C'est le seul site du crate qui *itère* une HashMap statique (les autres — QWERTY_AZERTY, HOMOGLYPHS, OCR_REPLACEMENTS, NICKNAMES_MAP — ne font que des `.get()`, sans problème).
- **Levier** : remplacer par un slice statique ordonné de tuples, comme `PHONETIC_MAP` le fait déjà dans `extra.rs:20-31` (`&[(&str, &str)]`).
- **Bit-identique** : **NON par rapport aux runs actuels** (qui ne sont de toute façon pas reproductibles entre eux sur ce chemin) — le fix *restaure* le déterminisme. À valider : deux runs successifs même seed → checksums identiques après fix (ce qui échoue probablement aujourd'hui sur un domaine à language_mix).
- **Priorité** : **P0 en tant que bug** (hors classement perf). Confiance : haute (comportement documenté de RandomState). Effort : trivial.
- **Mesure isolée** : avant tout fix, reproduire — petit run 2× même seed sur un domaine utilisant `addresses`, diff des sorties. Si identiques (par chance d'ordre stable dans ce build), le bug reste latent — fixer quand même.

### H20 — Pré-tirage RNG en `Vec<Vec<T>>` : 3-4 allocations heap par ligne

- **Fichiers** : `noise/typos.rs:116-129` (`apply_typos_str` : `ops`, `positions`, `rand_chars` en `Vec<Vec<_>>`), `:168-184` (`apply_typos_aggressive`, 5 tableaux), `:221-233` (`apply_typos_extreme`), `:276-278` (`apply_qwerty_azerty`) ; `noise/visual.rs:70-72` (`apply_homoglyph`), `:108-117` (`apply_unicode_pollution`), `:151-153` (`apply_ocr_errors`).
- **Statut** : vivant sur le chemin dups (chaque type de bruit typo/visual, par colonne ciblée).
- **Constat** : chaque tableau pré-tiré est un `Vec<Vec<usize>>` — une allocation heap par ligne et par tableau (jusqu'à 4-5 tableaux), soit ~4n petites allocs pour n lignes de dups, alors que les dimensions internes sont constantes (2, 3, 5 ou 8).
- **Levier** : aplatir en `Vec<usize>` de taille `n * k` indexé `[i * k + j]`. L'ordre des tirages RNG est inchangé (row-major dans les deux formes : `(0..n).flat_map(|_| (0..k))` ≡ boucles imbriquées actuelles).
- **Bit-identique** : **garanti par construction** (même séquence de tirages, mêmes valeurs consommées aux mêmes endroits). Ne PAS aller jusqu'au tirage inline dans la boucle de lignes : ça réordonnerait les tirages entre tableaux → sortie différente.
- **Priorité** : P2. Confiance : haute. Effort : faible (mécanique, 7 fonctions).

### H21 — `apply_unicode_pollution` : 31 tirages RNG et un Vec de 31 par ligne pour en utiliser 3
